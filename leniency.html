hanzi-writer の leniency 判定ロジック

対象バージョン hanzi-writer 3.7.3。options.leniency がストロークマッチャーの内部 5 チェックにどう作用するかをソース (dist/index.esm.js) ベースに整理した文書。kakitori (@k1low/kakitori) はこの値をそのまま forward しているだけなので、挙動の理解はそのまま kakitori の char.create({ leniency }) / block.create({ leniency }) / page.create({ leniency }) にも当てはまる。

TL;DR

項目内容
役割5 個ある判定閾値のうち 4 個を倍率でスケールする「マッチャーの寛容度」。デフォルト 1.0。
大きいほど各閾値が緩む / 受理しやすい (ただし方向チェックには無影響)。
小さいほど厳しい。0 以下や非有限値はマッチャーを silent に壊すので kakitori 側で reject。
適用箇所平均距離・始終点距離・形状 (Fréchet)・長さ比 の 4 チェック。方向 (cosine) のみ非依存。
追加効果後続ストロークの方が一致度が高いとき、自動で leniency を × 0.3〜0.6 引き締めて再チェック。

座標系 (1024 内部座標で判定、表示 size 非依存)

マッチャーは描画 size に依存しない。すべての距離 / 形状チェックは hanzi-writer の内部 1024 座標系で行われる。 median は元データ (animCJK / hanzi-writer-data-jp) が最初から 1024 系で持っており、ユーザーが描いた点は描画直後に Positioner.convertExternalPoint で 1024 系に逆射影されてから matcher に渡る。

CHARACTER_BOUNDS = [{x:0, y:-124}, {x:1024, y:900}] で character region は 1024 × 1024 (y は -124 .. 900 と非対称、ベースライン y=0 を含む)。 Positioner はこれを表示矩形にフィットさせる scale を計算する。

scale = min(displayWidth / 1024, displayHeight / 1024)

convertExternalPoint(p):
  x = (p.x - xOffset) / scale
  y = (height - yOffset - p.y) / scale   // Y up に反転

display 80px で描いた点も display 280px で描いた点も、同じ (512, 512) 付近に投影される。 その後の getAverageDistance 等が見るのはすべて 1024 系の値なので、閾値 (350, 250, 0.4, 0.35) はキャラクタ全体に対する 相対値として常に同じ意味になる。

閾値値単位size 依存
averageDistanceThreshold3501024 系の絶対距離 (= 1024 の約 34%)なし
START_AND_END_DIST_THRESHOLD2501024 系の絶対距離 (= 1024 の約 24%)なし
FRECHET_THRESHOLD0.4normalizeCurve で [0,1] に正規化された後の値なし (normalize で除去)

関連。 CharResult.perStroke[].points も同じ 1024 系で返るので、保存した結果を char.restore でどの size に展開しても、当時の描画を比例だけで再現できる。site の sizing 章で説明している不変条件と同じ。

マッチャーの全体フロー

ユーザーが 1 ストローク引き終わると strokeMatches(userStroke, character, strokeNum, options) が呼ばれる。内部の getMatchData() が下の AND ゲートを評価して、現ストロークについて isMatch を決める。

flowchart TD Start([User stroke 終了]) --> StripDup[stripDuplicates
連続重複点を除去] StripDup --> LenCheck{points.length
≥ 2 ?} LenCheck -- No --> Reject1([isMatch = false]) LenCheck -- Yes --> Match[getMatchData
現ストロークに対し AND 5 チェック] Match --> Backward{isMatch?
かつ checkBackwards} Backward -- isMatch=true --> Conflict[後続ストローク群を走査] Backward -- isMatch=false かつ checkBackwards --> Rev[points を逆順で再判定] Rev --> RevRes{逆順 isMatch?} RevRes -- Yes --> RejectBack([isMatch=false
meta.isStrokeBackwards=true]) RevRes -- No --> Reject2([isMatch = false]) Conflict --> ConflictCheck{closer match
found?} ConflictCheck -- No --> Accept([isMatch = true]) ConflictCheck -- Yes --> Tight[leniency × 0.3〜0.6
で再チェック] Tight --> AcceptOrReject([isMatch を最終決定])

5 つの AND チェック

以下のすべてが true のときだけ isMatch = true。各チェックに対する leniency の効き方を leniency 影響あり / 非依存 で示す。

1. 平均距離 (average distance) leniency 影響あり

ユーザーが描いた点の集合の各点について、対象ストローク median の最近傍点までの距離を取り、その平均を avgDist とする。これがしきい値以下なら通過。

avgDist ≤ averageDistanceThreshold × distMod × leniency
averageDistanceThreshold = 350 (デフォルト)
distMod = (isOutlineVisible || strokeNum > 0) ? 0.5 : 1

distMod は「2 ストローク目以降」または「outline 表示中」のときに 0.5 (= より厳しい) になる。最初の 1 ストロークだけ、outline 無しなら緩いという非対称な設計。
leniency は最終的な許容距離を線形に伸縮させる。leniency = 2.0 なら 2 倍まで距離があっても通る。

2. 始点 / 終点の距離 (start & end distance) leniency 影響あり

ユーザーストロークの始点と終点が、それぞれ対象 median の始点・終点に十分近いか。書き順だけでなく書き始めの位置も合わせさせるためのチェック。

startingDist ≤ START_AND_END_DIST_THRESHOLD × leniency &&
endingDist ≤ START_AND_END_DIST_THRESHOLD × leniency
START_AND_END_DIST_THRESHOLD = 250

leniency がそのまま許容距離倍率。0.5 なら 125、2.0 なら 500 まで OK。
平均距離は通っても始終点がズレていれば落ちる ("中央で形は合うが書き始めが違う" を弾く)。

3. 方向 (direction / cosine similarity) leniency 非依存

ユーザーが描いた各エッジベクトル (連続する点の差分) について、median 側の全エッジベクトルとの cosine similarity の最大値を取り、それらの平均が 0 を超えるか。

avgSimilarity = average(
  userEdges.map(e => max(strokeEdges.map(s => cos(s, e))))
)
pass ⇔ avgSimilarity > 0

leniency 非依存。閾値が 0 の固定値で、cosine が負 (= 90°超ズレ) でない限り通る非常に緩い判定。
ただし「右下に向かう median を、ユーザーが左上に書いた」のような 逆向きはここで検出される (次節の backward 再チェックに繋がる)。

4. 形状 (Fréchet 距離) leniency 影響あり

median とユーザーストロークそれぞれを正規化 (normalizeCurve) し、median を 5 段階の角度で回転 ( ±π/16, ±π/32, 0 ) させたうえで離散 Fréchet 距離を計算。最小値が閾値以下なら通過。

minDist = min over θ ∈ {0, ±π/32, ±π/16}
    frechetDist(norm(user), rotate(norm(median), θ))
pass ⇔ minDist ≤ FRECHET_THRESHOLD × leniency
FRECHET_THRESHOLD = 0.4

「形が似ているか」のチェック。Fréchet 距離は曲線の「最大隔たり」を見るので、平均距離は OK でも一部だけ大きく逸脱した曲線を弾ける。
微小回転を許容するのは「縦線をやや傾けて書いた」程度を吸収するため。

5. 長さ比 (length ratio) leniency 影響あり

ユーザー線分長と median 線分長の比 (両方に +25 のバイアスをかける) が 0.35 以上か。短すぎるストロークを弾く。

pass ⇔ leniency × (length(user) + 25) / (length(stroke) + 25) ≥ 0.35
MIN_LEN_THRESHOLD = 0.35

バイアス +25 は分母分子両方に乗るので、本来の median が非常に短い (点に近い) ときの計算が破綻しないよう緩衝する。
leniency が大きいほど「短いストロークでも通る」。leniency = 3.0 なら理論上は median の 1/8 強の長さでも通る。

Backward (逆向き) 検出

getMatchData が現ストロークで isMatch = false を返した場合、checkBackwards: true なら user points を逆順にして再度 getMatchData を呼ぶ。逆順だと一致する場合、isMatch はあくまで false のままだが meta.isStrokeBackwards = true をセットして返す。

これにより、上位 Quiz 層 (kakitori でいう char.mount のコールバック) は「向きが逆だっただけ」をユーザーに知らせたり、オプションの acceptBackwardsStrokes: true なら正解扱いに格上げしたりできる。

kakitori との関係。 hanzi-writer の acceptBackwardsStrokes は kakitori でも option として通っていて、デフォルト false。leniency を上げただけでは逆書きは通らない。

後続ストロークとの衝突解決

現ストローク strokeNum で AND 5 チェックを通っても、まだ書いていない後続ストローク strokeNum+1 .. n-1 をすべてスキャンし、それぞれ checkBackwards: false で getMatchData を回す。もし後続のどれかがより低い平均距離でマッチすると判明したら、ユーザーが順番を間違えた疑いがあるとみて、leniency を引き締めて現ストロークだけを再判定する。

closestMatchDist = min(後続ストローク群の avgDist)
if closestMatchDist < avgDist:
  leniencyAdjustment = 0.6 × (closestMatchDist + avgDist) / (2 × avgDist)
  # leniencyAdjustment は実質 [0.3, 0.6]
  再チェック: getMatchData(user, strokes[strokeNum], { leniency: leniency × leniencyAdjustment })
状況closestMatchDist / avgDistleniencyAdjustment意味
後続が同程度1.00.60軽い引き締め
後続がやや有利0.50.45中程度
後続がはるかに有利0 に近づく→ 0.30最大引き締め

つまり「あなたが今書いたのは、後で出てくる別ストロークの方がもっと似ている」というケースで、自動的に leniency × 0.3〜0.6 相当の厳しさで現ストロークを再評価する。元の leniency が 3.0 でも、衝突時の実効値は 0.9〜1.8 まで下がる。

kakitori との対応

kakitori 0.15.x は hanzi-writer のオプションを薄くラップしているだけ。leniency はそのまま HanziWriter.create(..., { leniency }) に forward される (packages/core/src/char.ts:1948)。

kakitori 側hanzi-writer 側
char.create(ch, { leniency }){ leniency } をそのまま forward
block.create({ leniency }) (block-wide)block 内の各 guided cell に char.create({ leniency }) として配る
page.create({ leniency }) (page-wide)page 内の各 block に BlockCreateOptions.leniency として配る
cells[i].overrides.leniency当該 guided cell の char.create 時に block/page wide より優先
computeSimilarity() 内部の HW_AVERAGE_DISTANCE_THRESHOLD = 350hanzi-writer の averageDistanceThreshold 既定値を mirror。マッチャーの再実行なしに similarity を [0,1] へ正規化するため

kakitori が独自に足したガードは:

  • leniency ≤ 0 / 非有限値の reject (block.create, page.create 入口、および cells[i].overrides.leniency)。hanzi-writer 側はこれらの値で silent に壊れる ( 全ストローク不一致 / 全部一致 など ) ので、kakitori は actionable なエラーに変換する。
  • blockwide / pagewide の precedence (per-cell overrides > block-wide > page-wide)。
